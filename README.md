## 1. 개요

### 1.1 목적
AI 기반 대화형 시스템에서 장기 기억을 효율적으로 관리하고 검색할 수 있는 계층적 의미 기억 시스템입니다. 기존의 선형적 기억 저장 방식의 한계를 극복하고, 의미적 연관성에 기반한 계층적 구조를 통해 더욱 효율적이고 맥락적인 기억 관리를 실현합니다. 대화형 시스템뿐만 아니라 다양한 문서 기록 시스템에서도 사용될 수 있다고 생각합니다. 다만 기존 검색 시스템보다 유연성이 필요한 검색 시스템에서 더 강점을 보여줍니다. 특정 단어나 문장을 명확하게 검색하는 시스템이 아니라. 비유된 표현, 바뀐 표현, 애매한 단어 묘사를 통한 검색에서 사용되기에 적절합니다.

### 1.2 시스템 아키텍처
```
┌─────────────────────────────────────────────────────┐
│                 User Interface                      │
│              (CLI, Discord Bot)                     │
└─────────────────┬───────────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────────┐
│                  MainAI                             │
│           (Main Conversation Handler)               │
└─────────────────┬───────────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────────┐
│                AuxiliaryAI                          │
│         (Assistant AI Classification)               │
└─────────────────┬───────────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────────┐
│              MemoryManager                          │
│         (Hierarchical Memory Manager)               │
└─────────────────┬───────────────────────────────────┘
                  │
┌─────────────────▼───────────────────────────────────┐
│  MemoryNode   │   DataManager  │ TreeCleanupEngine  │
│ (Memory Node) │ (Data Storage) │ (Tree Optimizer)   │
└─────────────────────────────────────────────────────┘
```

### 1.3 주요 특징
- **계층적 트리 구조**: 대화를 의미적 카테고리로 분류하여 계층적으로 관리합니다.
- **AI 기반 심층 탐색**: Gemini API를 활용하여 자동 대화 분류 및 정밀한 계층적 기억을 검색합니다.
- **비동기 병렬 처리**: 다중 API 키를 활용하여 여러 AI 작업을 동시에 처리함으로써 응답 속도를 향상시킵니다.
- **동적 트리 관리**: 대화가 누적됨에 따라 실시간으로 트리 구조를 최적화하고, 유사 카테고리를 자동으로 그룹화 및 병합합니다.
- **다중 인터페이스**: CLI(명령줄)와 Discord 봇 두 가지 방식으로 시스템을 사용할 수 있습니다.
- **동적 설정 변경**: CLI 대화형 모드에서 `!` 명령어를 통해 실시간으로 AI의 동작을 제어할 수 있습니다.

```

│   ├── __init__.py                 # 패키지 초기화
│   ├── AIManager.py                # AI 호출 관리자
│   ├── AuxiliaryAI.py              # 보조 AI (분류/검색 담당)
│   ├── DataManager.py              # 데이터 관리자
│   ├── MainAI.py                   # 메인 AI (대화 총괄)
│   ├── MemoryManager.py            # 기억 관리자
│   ├── MemoryNode.py               # 기억 노드 클래스
│   └── TreeCleanupEngine.py        # 트리 정리 엔진
├── memory/                         # 기억 저장소
│   ├── all_memory.json             # 전체 대화 원본 기록
│   └── hierarchical_memory.json    # 계층적 트리 구조
├── config.py                       # 설정 및 Few-shot 데이터
├── main.py                         # 메인 실행 파일 (CLI)
├── hsms_discord.py                 # Discord 봇 실행 파일
├── requirements.txt                # 패키지 의존성
└── README.md                       # 프로젝트 문서
```

### 3.1 데이터 파일
- **`memory/all_memory.json`**: 모든 대화의 원본이 순차적으로 저장되는 파일입니다. 각 대화는 고유한 인덱스를 가집니다.
- **`memory/hierarchical_memory.json`**: 대화의 의미적 관계를 나타내는 트리 구조가 저장됩니다. 각 노드는 `all_memory.json`에 저장된 대화의 인덱스를 참조합니다.

## 4. 핵심 모듈 분석

### 4.1 AIManager.py (AI 호출 관리자)
- AI 호출 최적화, 비동기 병렬 처리, 오류 처리 및 성능 모니터링을 담당합니다.
- `call_ai_async_multiple`: 다중 API 키를 사용하여 여러 AI 요청을 동시에 처리하여 응답 속도를 극대화합니다.

### 4.2 AuxiliaryAI.py (보조 AI)
- 시스템의 핵심 두뇌로, 대화 분류 및 기억 검색을 수행합니다.
- `handle_conversation`: 새로운 대화를 받아 적절한 카테고리를 찾거나 생성하고, 트리 구조에 맞게 저장합니다.
- `search_relevant_memories`: 사용자의 질문을 받아 계층적 심층 탐색 알고리즘을 통해 가장 관련성 높은 기억을 찾아냅니다.

### 4.3 DataManager.py (데이터 관리자)
- `memory` 폴더의 JSON 파일에 대한 읽기/쓰기를 담당합니다.
- 파일 손상 시 자동 백업 및 복구 기능을 포함합니다.

### 4.4 MainAI.py (메인 AI)
- 사용자와의 상호작용을 총괄하는 최상위 컨트롤러입니다.
- 사용자의 입력을 받아 기억 검색이 필요한지 판단하고, `AuxiliaryAI`를 통해 얻은 기억을 바탕으로 최종 응답을 생성합니다.
- 대화형 모드의 동적 명령어 처리를 위한 상태(force_search, no_record 등)를 관리합니다.

### 4.5 MemoryManager.py (기억 관리자)
- `hierarchical_memory.json` 파일을 관리하며, 기억 트리의 구조(노드 추가, 수정, 삭제 등)를 직접 제어합니다.

### 4.6 MemoryNode.py (기억 노드)
- 기억 트리를 구성하는 각 노드를 정의하는 클래스입니다.
- `topic`, `summary`, `parent_id`, `children_ids`, `conversation_indices` 등의 속성을 가집니다.

### 4.7 TreeCleanupEngine.py (트리 정리 엔진)
- `--clean-tree` 명령 실행 시 호출되어 누적된 기억 트리를 최적화합니다.
- 유사한 카테고리나 대화 노드를 자동으로 병합하고, 너무 많은 자식 노드를 가진 카테고리를 그룹화하여 트리 구조를 건강하게 유지합니다.

## 5. 실행 방법

### 5.1 대화형 모드 (CLI)

가장 기본적인 실행 방법입니다. 터미널에서 사용자와 AI가 대화를 주고받습니다.

```bash
python main.py --mode chat
```

#### 5.1.1 대화 중 명령어 사용

대화형 모드 실행 중 `!`로 시작하는 명령어를 입력하여 실시간으로 AI의 동작을 제어할 수 있습니다.

- **`!help`**: 사용 가능한 모든 명령어를 표시합니다.
- **`!status`**: 현재 설정 상태(ON/OFF, 변수 값 등)를 보여줍니다.
- **`!force-search`**: 모든 대화에서 기억을 강제로 검색하는 모드를 켜거나 끕니다.
- **`!force-record`**: AI 응답 없이 대화를 기록만 하는 모드를 켜거나 끕니다. (`!no-record`와 동시 사용 불가)
- **`!no-record`**: 대화를 아예 저장하지 않는 순수 응답 모드를 켜거나 끕니다. (`!force-record`와 동시 사용 불가)
- **`!debug`**: 상세한 처리 과정을 보여주는 디버그 모드를 켜거나 끕니다.
- **`!tree`**: 현재 기억 트리 구조를 출력합니다.
- **`!api-info`**: 설정된 API 키 정보를 보여줍니다.
- **`!max-depth [숫자]`**: 기억 트리의 최대 깊이를 설정합니다. (예: `!max-depth 5`)

### 5.2 기타 실행 모드

```bash
# 테스트 모드 (미리 정의된 질문들로 시스템 자동 테스트)
python main.py --mode test

# 검색 전용 모드 (기억 탐색만 수행, 저장 안 함)
python main.py --mode search

# Discord 봇 모드
python main.py --mode discord
```

### 5.3 명령줄 인수 상세 설명

프로그램 시작 시 아래 인수를 사용하여 특정 모드로 실행할 수 있습니다.

#### 5.3.1 동작 제어
- `--force-search`: (초기 설정) 모든 대화에서 강제로 기억 검색
- `--force-record`: (초기 설정) AI 응답 생성 없이 대화만 기록
- `--no-record`: (초기 설정) 대화를 기억 시스템에 저장하지 않음
- `--debug`: (초기 설정) 상세한 디버그 정보 출력

#### 5.3.2 트리 구조 설정
- `--max-depth N`: 트리 최대 깊이 설정 (기본값: 4, 최소: 3)
- `--top-search-n N`: 검색 시 반환할 최대 대화 수 (0=무제한)

#### 5.3.3 트리 관리 기능
- `--clean-tree`: 트리 구조 최적화 실행
- `--fanout-limit N`: 노드당 최대 자식 수 (기본값: 12)
- `--rename`: 트리 정리 시 노드 이름 자동 재생성
- `--dry-run`: 실제 변경 없이 최적화 계획만 출력

#### 5.3.4 정보 조회
- `--tree`: 현재 트리 구조 시각화 출력
- `--api-info`: 설정된 API 키 정보 표시

## 7. 설정 파일 분석 (`config.py`)

### 7.1 주요 상수
- **`ALL_MEMORY`**, **`HIERARCHICAL_MEMORY`**: 기억 데이터 파일 경로
- **`GEMINI_MODEL`**: 사용할 Gemini 모델 버전
- **`API_KEY`**, **`LOAD_API_KEYS`**: `.env` 파일에서 불러온 API 키

### 7.2 Few-shot 데이터 세트
Gemini API의 `system_instruction`만으로는 일관된 판단을 보장하기 어려울 때, 구체적인 예시를 제공하여 AI의 응답을 유도하는 데이터입니다.
- **`NODE_SIMILARITY_FINE`**: 두 노드가 유사한지 판단하는 예시
- **`CATEGORY_RELEVANCE_FINE`**: 대화와 카테고리의 관련성을 판단하는 예시
- **`MEMORY_SEARCH_FINE`**: 기억 검색이 필요한지 판단하는 예시
- ... 등 다양한 판단 기준에 대한 예시 데이터 포함

## 8. 기능별 상세 분석

### 8.1 계층적 기억 구조

#### 8.1.1 트리 구조 개념
- **ROOT 노드**: 전체 트리의 최상위 노드입니다.
- **카테고리 노드**: 하위 노드들을 그룹화하는 중간 노드입니다. (예: '과학', '취미')
- **대화 노드 (리프 노드)**: 실제 대화 내용의 요약과 원본 대화 인덱스를 가지는 최하위 노드입니다. (예: '양자역학', '클래식 음악 감상')

### 8.2 AI 기반 분류 시스템

#### 8.2.1 분류 과정
1.  **관련성 평가**: 사용자 입력과 기존 최상위 카테고리들의 관련성을 AI가 병렬로 판단합니다.
2.  **카테고리 수에 따른 분기**:
    - **관련 카테고리 없음**: AI가 새로운 카테고리명과 요약을 생성하여 새 노드를 만듭니다.
    - **관련 카테고리 1개**: 해당 카테고리 내에서 다시 하위 노드들과의 관련성을 판단하여 대화를 추가하거나 새 하위 노드를 생성합니다.
    - **관련 카테고리 2개 이상**: AI가 두 카테고리의 유사도를 판단하여 하나의 상위 그룹으로 묶을지, 아니면 가장 관련성 높은 카테고리 하나를 선택할지 결정합니다.

### 8.3 기억 검색 시스템

#### 8.3.1 검색 필요성 판단
AI가 사용자 입력을 분석하여 과거 기억 참조 필요성을 판단합니다.
- **True**: 과거 대화 참조 필요 (예: "저번에 말한 그 영화 제목이 뭐였지?")
- **False**: 일반적인 질문 (예: "오늘 날씨 어때?")

#### 8.3.2 계층적 심층 탐색 알고리즘
단순히 첫 번째 계층만 탐색하는 방식에서 벗어나, 너비 우선 탐색(BFS)과 유사한 방식으로 트리를 계층별로 정밀하게 탐색합니다.

1.  **1단계 (최상위 카테고리 평가)**: 루트 노드의 직계 자식(최상위 카테고리)들의 관련성을 먼저 평가합니다.
2.  **2단계 (심층 탐색)**: 1단계에서 관련 있다고 판단된 카테고리의 **하위 노드들만**을 대상으로 다시 관련성을 평가합니다. 이 과정을 트리의 끝(리프 노드)에 도달할 때까지 반복합니다.
3.  **3단계 (리프 노드 수집)**: 탐색 과정에서 최종적으로 관련 있다고 판단된 **리프 노드(가장 하위의 대화 노드)**의 대화 내용만을 결과에 포함시킵니다.

이 방식을 통해 관련 없는 상위 카테고리에 속한 수많은 대화들을 사전에 필터링하여, 최종적으로는 질문과 가장 밀접한 핵심 기억만을 정확하게 찾아낼 수 있습니다.

### 8.4 트리 최적화 시스템 (`--clean-tree`)

#### 8.4.1 최적화 필요성
- **과도한 팬아웃**: 한 노드의 자식이 너무 많아 구조 파악이 어려운 경우
- **중복/유사 노드**: 이름이 같거나 의미적으로 매우 유사한 노드가 여러 개 존재하는 경우

#### 8.4.2 클러스터링 및 병합 알고리즘
1.  **유사도 매트릭스 구축**: AI를 활용하여 모든 노드 쌍의 의미적 유사도를 계산합니다.
2.  **클러스터링**: 유사도 매트릭스를 기반으로 비슷한 노드들을 그룹(클러스터)으로 묶습니다.
3.  **자동 그룹화 및 병합**:
    - **자식 과다 노드**: 자식 노드들을 클러스터링하여 새로운 중간 그룹 노드를 생성하고 재배치합니다.
    - **유사 카테고리**: 이름이 다르더라도 의미가 비슷한 카테고리들을 찾아 하나의 대표 카테고리로 병합합니다.
    - **유사 리프 노드**: 비슷한 내용을 담고 있는 최하위 대화 노드들을 병합합니다.
    - **중복 노드**: 이름이 완전히 같은 노드들을 병합합니다.
